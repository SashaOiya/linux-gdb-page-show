# Page Table for GDB & QEMU

Данный проект представляет собой инструмент для инспекции и визуализации таблиц страниц пользовательского пространства (User Space Page Tables) в операционной системе Linux, запущенной на эмуляторе архитектуры **RISC-V**.

В состав проекта входит кастомный скрипт автоматизации для GDB на Python, который обходит баги стандартного интерфейса чтения памяти GDB, запрашивая данные напрямую у физической памяти QEMU через отладочный монитор. Это позволяет полностью развернуть структуру аппаратных страниц `PGD -> PMD -> PTE` вплоть до конечных фреймов физической RAM.

---

## 🛠️ Архитектура и компоненты

* `init.c` — Минималистичный пользовательский процесс (userspace init), написанный на Си. Служит точкой входа для операционной системы и используется для демонстрации ленивой аллокации страниц (`Lazy/CoW allocation`).
* `build.sh` — Скрипт автоматизации верхнего уровня. Отвечает за скачивание, конфигурацию и сборку самого ядра Linux и таргетного загрузчика **OpenSBI** под архитектуру RISC-V.
* `run.sh` — Скрипт для запуска эмулятора QEMU с поддержкой GDB-сервера (порт `:1234`) в режиме ожидания отладчика.
* `linux_gdb_page_show.py` — Расширение для GDB, добавляющее команду `lx-user-pt <PID>` для визуализации дерева страниц.

---

## 🚀 Инструкция по сборке и запуску

### 1. Сборка системного окружения (Linux + OpenSBI)

Скрипт `build.sh` подготавливает всю низкоуровневую подсистему. Запусти его один раз для сборки ядра:

```bash
chmod +x build.sh run.sh
./build.sh

```

### 2. Компиляция тестового процесса `init.c`

Пользовательский бинарник компилируется отдельно с помощью кросс-компилятора `riscv64-linux-gnu-gcc`. Обязательно используется флаг `-static`, чтобы исключить зависимости от динамических библиотек (так как наш `init` запускается в изолированной пустой системе):

```bash
riscv64-linux-gnu-gcc -static init.c -o init

```

*Полученный бинарник `init` помещается в образ диска / initramfs перед запуском системы.*

### 3. Запуск эмулятора QEMU

Скрипт `run.sh` запускает собранное ядро Linux и OpenSBI внутри QEMU с флагами `-S` (остановить процессор на старте) и `-s` (открыть порт `1234` для GDB).

Запусти эмулятор в отдельном терминале:

```bash
./run.sh

```

*Эмулятор успешно запустится и замрет в ожидании подключения отладчика.*

## 🔍 Инструкция по запуску визуализации

Чтобы запустить трассировку таблиц страниц и увидеть дерево маппинга памяти, выполни следующие шаги в терминале:

1. **Подключись через GDB к эмулятору:**
```text
$ ./run.sh

```
2. **Делаем continue:**
```text
  (gdb) c

```
Ждём секунд 10, потом делаем Ctrl+C

3. **Запусти визуализацию:**
Теперь можем вывести дерево страниц для **PID 1** (наш процесс `init`):

```text
   (gdb) lx-user-pt 1

```

## 📊 Как устроена визуализация (Принцип работы)

Скрипт разбирает структуру виртуальной памяти согласно спецификации трехуровневой архитектуры **RISC-V**:

При вызове команды происходят следующие этапы:

1. **Поиск дескриптора:** Скрипт находит структуру `task_struct` для указанного PID, заглядывает в поле `mm_struct` и вытаскивает виртуальный адрес корня таблицы страниц (`pgd`).
2. **Прямой доступ к RAM:** Скрипт переводит этот адрес в физический и делает низкоуровневый запрос к эмулятору через команду `monitor xp`. Это полностью исключает баги знакового расширения 64-битных адресов ядра в GDB Python API.
3. **Рекурсивный обход уровней:** Скрипт поочередно спускается по цепочке `PGD (Page Global Directory) -> PMD (Page Middle Directory) -> PTE (Page Table Entry)`, декодируя каждый физический указатель.

### Пример успешного вывода на уровне PTE:

```text
Построение дерева страниц для процесса: init (PID: 1)
PGD (корень) Физический адрес: 0x822cb000
------------------------------------------------------------
└── PGD [000] -> Переход на PMD (Физический адрес таблицы: 0x822d2000)
    └── PMD [000] -> Переход на PTE (Физический адрес таблицы: 0x822d0000)
        └── PTE [000] -> Физическая RAM: 0x822cf000 [S-R--] [Lazy/CoW allocation] (Обычная страница (4 КБ))
    └── PMD [255] -> Переход на PTE (Физический адрес таблицы: 0x822d3000)
        └── PTE [510] -> Физическая RAM: 0x822cd000 [S-R--] [Lazy/CoW allocation] (Обычная страница (4 КБ))
        └── PTE [511] -> Физическая RAM: 0x822d4000 [S-R--] [Lazy/CoW allocation] (Обычная страница (4 КБ))

```

* **`PMD [000]`** — Секция, где живет исполняемый код бинарника `init` (сегмент `.text`).
* **`PMD [255]`** — Секция пользовательского стека процесса. Так как в Linux стек растет сверху вниз, данные занимают самые последние индексы в таблице (`PTE [510]` и `[511]`).
* **`[Lazy/CoW allocation]`** — Индикатор механизма отложенной аллокации ядра Linux. Страницы уже замаппированы в таблицах, но аппаратно биты доступа `R/W/X` сброшены в 0. При первом физическом обращении процесса к этой памяти ядро обработает *Page Fault*, выдаст права, и страница станет активной.
